# SKILL: BPC Product System

## Purpose
Use this skill to structure products, modules, bundles, subscriptions, and customer progression for The Body Protection Company.

This skill governs:
- product hierarchy
- SKU logic
- module structure
- bundle logic
- store structure
- subscription logic
- product progression
- system architecture

## When to use
Use this skill whenever the task involves:
- product architecture
- designing a system or module
- naming a module
- deciding what a user buys
- building subscriptions
- building bundle logic
- store navigation or merchandising logic

Use alongside the BPC Agent Operating System skill.

## Core product principle
We do not sell compounds.

We sell:
STRUCTURED BIOLOGICAL SYSTEMS

Compounds are components.
Systems are the product.

## Product hierarchy
Always preserve:

SYSTEM → MODULE → COMPOUND → BATCH

### System
A complete biological objective.
Examples:
- Recovery System
- Metabolic System
- Hormonal System
- Cellular System

### Module
A focused sub-function within a system.
Examples:
- Tissue Repair Module
- Inflammation Control Module
- Fat Loss Module
- Glucose Regulation Module

### Compound
The actual peptide or input.
Examples:
- BPC-157
- TB-500
- CJC-1295
- Ipamorelin

### Batch
The trust unit.
Includes:
- batch ID
- HPLC results
- purity
- manufacturing origin

## SKU rules

### Primary SKU
System SKU
This is highest-level value.

### Main acquisition SKU
Module SKU
This should be the main entry point for users.

### Restricted / secondary SKU
Compound SKU
Only when context requires it.
Never position standalone compounds as the core business model.

## Naming rules
Prefer:
[FUNCTION] + SYSTEM or MODULE

Examples:
- Recovery System
- Tissue Repair Module
- Metabolic System
- Glucose Control Module

Do not use:
- hype names
- bro-science names
- consumer wellness naming

## Bundle logic
Structure offers as:
- single module entry
- multi-module stack
- full system bundle

Each should feel like increasing control and system depth.

## Progression logic
The user should move through stages:

1. Entry
Single module

2. Expansion
Add second module

3. Integration
Multi-module system

4. Dependence
BPC becomes default trusted input infrastructure

## Protocol requirement
No product exists without protocol context.

Every system or module should include:
- purpose
- mechanism
- usage structure
- stack logic
- expected outcome range
- verification access

## Subscription logic
Subscriptions should reinforce:
- consistency
- continuity
- reliability
- input control

Do not frame subscriptions as generic autoship.

Frame them as maintaining structured input control.

## Store / merchandising rules
Do not organize the store by compounds first.

Do organize by:
SYSTEM → MODULE

Wrong:
- BPC-157
- TB-500
- CJC-1295

Correct:
- Recovery System
- Tissue Repair Module
- Inflammation Module

## Trust layer
Every product should expose:
- batch ID
- purity data
- HPLC access
- origin
- transparent verification

Trust must be visible, immediate, and unavoidable.

## Constraints
Do not:
- reduce products into SKUs without system logic
- merchandise like a peptide commodity site
- make compounds the hero
- create random bundles without progression logic

## Output instructions
When asked to design or critique a product structure, include:
- system
- module
- compounds involved
- user entry point
- progression path
- subscription potential
- trust layer
- why this fits BPC doctrine

## Final standard
BPC is not a peptide store.

It is a structured biological system.

If the output feels like a product catalog instead of a system, it is wrong.
